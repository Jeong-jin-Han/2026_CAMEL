# BeaconGNN: Large-Scale GNN Acceleration with Out-of-Order Streaming In-Storage Computing

> **HPCA 2024** · `cluster/isc` · Source: [BeaconGNN - Large-Scale GNN Acceleration with Asynchronous In-Storage Computing.pdf](<BeaconGNN - Large-Scale GNN Acceleration with Asynchronous In-Storage Computing.pdf>)

저자: Yuyue Wang¹, Xiurui Pan², Yuda An², Jie Zhang²† (교신), Glenn Reinman¹† (교신) — ¹UCLA, ²Peking University

> 참고: 폴더/스텁 파일명은 "Asynchronous"이나 PDF p.1 실제 제목은 **"Out-of-Order Streaming In-Storage Computing"**. 동일 논문(BeaconGNN, HPCA 2024)이며 본 노트 제목은 PDF 원문을 따른다.

## TL;DR
대규모 GNN은 graph 구조와 feature table이 모두 TB급으로 storage에 상주하므로 data preparation(neighbor sampling + feature lookup) 단계에서 storage↔processor 데이터 전송이 병목이 된다. 기존 ISC-for-GNN(GList, SmartSage)은 (1) hop 단위 in-order 샘플링으로 flash 내부 병렬성을 못 살리고, (2) page보다 작은 random I/O를 page-granular channel 전송으로 처리해 대역폭을 낭비하며, (3) firmware-기반 I/O 처리가 ULL flash throughput을 못 따라간다. BeaconGNN은 (a) flash physical address를 graph에 직접 박은 **DirectGraph** 포맷으로 hop 간 host-SSD 통신과 주소 변환을 제거하고, (b) **die-level sampler / channel-level command router / controller-level spatial accelerator**의 3단 near-data 엔진으로 out-of-order 스트리밍 처리를 한다. 결과: state-of-the-art ISC(BG-DGSP) 대비 throughput 최대 11.6×, 에너지 효율 4×; CPU-centric baseline 대비 27.3×.

## 문제 & 동기
GNN task는 *data preparation*(graph에서 multi-hop neighbor 샘플링 → feature table에서 vector fetch)과 *GNN computation*(aggregation/update)으로 나뉜다. 실세계 산업 그래프는 수백 GB~TB라 single machine 메모리를 초과해 SSD에 상주한다. 그러면 data preparation이 thin PCIe bus를 사이에 둔 대량 전송을 유발하고, storage software stack(file system + NVMe)이 redundant copy와 다중 주소 변환을 추가한다. ISC와 ULL flash가 기회를 주지만, 기존 ISC-for-GNN에는 세 가지 근본 결함이 있다.

> [!quote]- 📄 원문 표현 (paper)
> "Prior in-storage computing (ISC) solutions show fundamental drawbacks when applied to GNN acceleration. First, they obey a strict ordering of GNN neighbor sampling. Such serialization fails to utilize flash internal parallelism. Second, the I/O sizes generated by GNN are much smaller than the minimum flash access granularity. The limited channel bandwidth is wasted when serving the requests. Third, the prior solutions rely on firmware-based request processing, making the backend I/O throughput constrained by the embedded core processing power." (Abstract, p.1)
>
> "the data preparation stage incurs intensive data transfer between storage and processors ... the thin PCIe bus narrows down the I/O bandwidth while the software stack of storage and GPUs introduces redundant data copies and multiple address translations" (p.1)
>
> "increasing active ULL-flash dies from 1 to 8 brings merely 49% throughput improvement but undertakes 7.7× average latency." (Challenge 2, p.4)

## 핵심 통찰

> [!note]- 통찰 토글
> 분석에서 도출한 세 통찰(p.2):
> 1. **Hop-by-hop ordering이 내부 병렬성을 망친다.** Data preparation을 control path에서 떼어내 non-intermittent ISC task로 통째로 돌리면 hop 간 barrier가 사라진다 → out-of-order sampling 가능.
> 2. **Page-level channel 전송은 small random I/O에 부적합.** Flash die의 control layer에는 여유 area budget이 있어([10]) unused 데이터를 걸러내는 추가 logic을 die에 둘 수 있다 → channel 전송 효율 개선.
> 3. **Firmware가 flash I/O를 못 따라간다.** Hardware-기반 customized flash I/O control로 처리를 가속해야 ULL throughput을 산다.
>
> 세 통찰이 각각 DirectGraph(통찰1), die-level sampler/channel router(통찰2·3), spatial accelerator(연산)로 매핑된다. 핵심은 "host control을 GNN preparation 전 과정에서 제거"하여 SSD 내부가 자율적으로 스트리밍하도록 만드는 것.

## 설계 / 메커니즘

> [!abstract]- 설계 토글
> **두 버전으로 점진 설계.**
> - **BeaconGNN-1.0 (BG-1)**: GList(feature table 오프로드)+SmartSage(graph 구조 오프로드)를 합쳐 전체 GNN task를 SSD로 오프로드 + ULL flash 적용한 baseline 조합. 단, 위 세 challenge가 그대로 남는다(host가 hop마다 node↔LPA 변환·다음 hop 명령). (Fig.4, p.4)
> - **BeaconGNN-2.0 (BG-2)**: 본 제안. DirectGraph + 다단 near-data 엔진으로 세 challenge 해결. (Fig.9, p.5)
>
> **(1) DirectGraph GNN format (§IV-A, Fig.8, p.5).** GNN 데이터는 거의 static이므로 매 hop마다 logical→physical 주소를 다시 번역할 필요가 없다는 관찰에서 출발. 그래프를 flash physical page에 정렬된 **primary page / secondary page**로 조직.
> - 각 node는 feature vector + neighbor list를 **primary section**에 저장. 넘치면 **secondary section**(secondary page)으로 확장.
> - low-degree node들의 primary section을 linked array로 묶어 한 primary page에 compaction(공간 낭비 방지).
> - 각 neighbor index는 indirection 없이 **4-byte physical address**로 매핑(1TB·4KB SSD에서 28bit page + 4bit in-page section, log₂(1TB/4KB)=28). → host는 mini-batch 시작 시 target node의 primary section 주소만 주고, 이후 모든 주소 지정은 SSD 내부에서 처리.
>
> **(2) 다단 near-data processing 엔진 (§V).**
> - **Die-level sampler (Fig.10·11, p.6)**: flash die control layer에 GNN config register + sampler + TRNG를 추가(two-plane 공유). 한 page를 cache register로 읽어 그 page 안에서 neighbor를 샘플링하고, 샘플 결과로 다음 sampling command를 생성해 data register에 기록. primary section은 feature vector도 retrieve, final hop은 샘플링 중단. sampler 4구성요소: section iterator / vector retriever / node sampler / command generator. 샘플링은 TRNG + modulo로 random.
> - **Channel-level command router (Fig.12, p.6)**: flash interface controller에 inter-channel router 추가. die가 만든 sampling command를 channel data stream에서 parser가 추출 → crossbar로 destination channel에 라우팅(embedded processor 개입 없음). dispatch queue(die별) + round-robin command issuer + DMA(feature vector 전송, DRAM 주소/size 재설정 불필요)로 ULL throughput 확보.
> - **Bus-attached spatial accelerator (§V-C, p.7)**: SSD 내부 bus에 부착. 1D vector array(feature aggregation) + 2D systolic array(GEMM-based update) + 공유 SRAM buffer. flash backend 데이터는 먼저 SSD DRAM에 쓰인 뒤 accelerator로.
>
> **(3) System support (§VI).** firmware가 DirectGraph용 physical block 목록을 예약(ioctl NVMe 명령으로 노출), 일반 FTL 우회·일반 alloc/GC에서 제외. DirectGraph construction은 Algorithm 1(2단계: metadata 수집 → page 직렬화 후 SSD flush). ONFI 명령 2개 customize(global GNN config / sampling). firmware-based GNN engine이 mini-batch 단위로 workflow 스케줄링하며 **이전 mini-batch의 computation과 현재 mini-batch의 data preparation을 pipeline overlap**.
> - **Reliability(§VI-F)**: idle 시 DirectGraph block을 주기적으로 data scrubbing(ECC로 검사, 에러 발견 시 block 전체 erase·재기록). SLC Z-NAND(RBER<10⁻⁷) 사용. DirectGraph block은 일반 wear leveling에서 빠지므로 P/E 불균형이 임계 넘으면 reclamation으로 마이그레이션.
> - **Security(§VI-E)**: DirectGraph block을 FTL에서 unusable로 표시해 host stack에서 격리. flushing/runtime에 destination·section 주소가 모두 할당 block 내인지 엄격 검증.
> - **두 모드**: regular-I/O / acceleration. acceleration 중 들어온 regular I/O는 현재 mini-batch 끝으로 deferred. page table은 SSD-DRAM에 상주(swap 없음)해 acceleration 중에도 regular request 빠르게 처리.

> **데이터셋 (Table III, p.8)**: Reddit, Amazon(265.9M node/79.8B edge), Movielens, OGBN, PPI. 3-hop·hop당 3 neighbor(target당 총 40 node), vector_sum aggregation + perception update, 128-dim FP16.
> **비교 플랫폼**: CC(CPU-centric), BG-1, BG-DG(BG-1+DirectGraph), BG-SP(BG-1+die sampler), BG-DGSP(DG+SP, = SOTA ISC), BG-2(DGSP+channel router).
>
> **Throughput (Fig.14, p.8~9).** CC 대비 평균: SmartSage 2.11×, GList 1.42×, BG-1 2.35×, BG-DG는 BG-1 대비 marginal(whole-page 전송이 chip 활용을 낮게 억제). **BG-SP는 BG-1 대비 5.47×**(sampling barrier 해소로 flash resource 활용↑), BG-DGSP는 거기서 +20%(out-of-order). **BG-2는 BG-DGSP 대비 +41%, 전체 21.70× speedup**(CC 대비).
> **전체 요약 수치**: 최대 **27.3× throughput, 11.6× overall throughput, 9.86×·4× energy efficiency** (CPU-centric baseline / SOTA ISC 대비, p.2).
> **Flash resource utilization (Fig.15, p.9~10).** BG-SP는 sampling barrier 지점에 3개의 두드러진 low-utilization valley(한 hop sampling이 끝나길 기다리며 idle). BG-DGSP는 처음부터 일관된 high utilization. **BG-2는 BG-SP·BG-DGSP 대비 channel·die 활용을 76% 증가, 총 sampling latency를 78% 감소**(firmware를 sampling path에서 제거). reddit/PPI는 고차원 feature로 channel 전송이 die 활용 초과; movielens/OGBN은 짧은 feature로 channel 활용 낮음. amazon이 대표적(channel·die 모두 high).
> **Latency breakdown (Fig.15f, amazon, p.9).** CC는 PCIe 전송이 큰 비중. BG-1은 PCIe 대부분 제거(flash 시간 최대). BG-DG→BG-2로 flash I/O latency 점감. host-side delay는 minor.
> **Command latency breakdown (Fig.17, p.10).** BG-DG·BG-DGSP는 wait_before_flash가 BG-1·BG-DG 대비 41%·42% 더 김(DirectGraph가 더 많은 command를 ready로 만들어 대기 누적). **BG-2는 hardware core 처리로 BG-DGSP 대비 wait time 68% 감소.**
> **Sensitivity (Fig.18, §VII-C, p.10~11).** batch size(32~256): BG-2가 가장 잘 scale, 수렴 없음. channel bandwidth(333/1600/2400 MB/s): BG-1·BG-DG는 page-granular 전송 때문에 크게 개선, BG-2는 800MB/s 이상에서 flash 효과 throughput 포화로 개선 미미. channel 수: BG-2는 16 channel까지 linear, 이후 DRAM 전송이 병목. die/channel: BG-2 16 dies/channel까지 linear. page size(2~16KB): BG-2 무관.
> **Energy (Fig.19, amazon, p.11~12).** CC는 storage 밖 전송에 57% 소비. BG-1·BG-DG는 whole page를 SSD DRAM으로 → 75%가 거기 소비. BG-SP/DGSP/BG-2는 그것 제거, flash backend 57% + DRAM·accelerator 43%. **BG-2 평균 power 13.4W로 PCIe 75W 한계 이하.**
> **Traditional SSD(20μs read-latency, §VII-E, p.12).** CC 대비 BG-1 2.20×, BG-2 2.50×, BG-SP 3.19×, BG-DGSP 4.19×, BG-2 4.19×. → DirectGraph+die sampling은 ULL뿐 아니라 traditional SSD에도 유효. 단 high read latency에선 BG-DGSP≈BG-2(channel router 불필요).
> **DirectGraph storage 효율 (Table IV, §VII-F, p.12).** inflate ratio: reddit 2.8%, amazon 4.1%, movielens 3.5%, PPI 3.5% — OGBN만 32.3%(평균 degree 28로 낮아 short section의 page 낭비; densification law상 medium-scale에서만 발생).

## 핵심 용어
- **ISC (In-Storage Computing)**: SSD 내부 프로세서/FPGA/ASIC로 연산을 수행해 PCIe 전송을 줄이는 기법.
- **ULL flash (Ultra-Low Latency flash)**: SSD read(sense) latency를 3μs까지 낮춘 flash(예: Z-NAND); 본 논문이 노리는 매체.
- **Data preparation**: GNN의 neighbor sampling + feature lookup 단계로, 대규모 그래프에서 I/O 병목의 주범.
- **Neighbor sampling / k-hop subgraph**: target node 기준 각 hop에서 일부 neighbor만 추려 작은 subgraph를 만드는 GraphSage 최적화.
- **DirectGraph**: neighbor index를 4-byte flash physical address로 직접 박고 primary/secondary section을 physical page에 정렬한 GNN 포맷; 다단 주소 변환과 hop barrier 제거.
- **Primary / secondary section(page)**: node의 feature+neighbor를 담는 기본 영역 / 넘칠 때 확장하는 영역. low-degree node는 primary를 compaction.
- **Inter-hop sampling barrier**: hop마다 host-SSD 통신으로 다음 hop을 시작해야 해 flash die 병렬성을 막는 직렬화 장벽(Challenge 1).
- **Page-granular channel transfer**: small random I/O를 page 단위로 전송해 대역폭을 낭비하는 문제(Challenge 2).
- **Die-level sampler**: flash die control layer에 둔 sampler(section iterator/vector retriever/node sampler/command generator + TRNG)로 page 내에서 샘플링·feature retrieve.
- **Channel-level command router**: flash interface controller에 둔 crossbar+dispatch queue로 sampling command를 firmware 없이 destination channel로 라우팅.
- **Out-of-order streaming ISC**: hop 순서 제약을 풀고 SSD 내부가 자율 스트리밍으로 샘플링·전송하는 본 논문의 패러다임.
- **Spatial accelerator**: SSD bus에 부착한 1D vector array(aggregation)+2D systolic array(GEMM update).
- **Inflate ratio**: 원본 그래프를 DirectGraph로 변환했을 때의 저장 공간 증가율.

## 강점 · 한계 · 열린 질문
**강점**
- 전체 GNN task flow(샘플링+lookup+연산)를 SSD 내부로 완전 오프로드한 first full-stage ISC, 세 challenge를 정확히 진단하고 각각에 대응하는 component를 1:1로 설계.
- DirectGraph로 hop barrier와 다단 주소 변환을 동시 제거, software-hardware co-design이 일관됨.
- area overhead 미미(router 1.26%, sampler <0.1%), power 13.4W로 PCIe 한계 내. traditional SSD에서도 유효(4.19×).
- reliability/security/wear-leveling을 별도 절로 다뤄 실용성 논거 보강.

**한계**
- 평가가 자체 Python cycle-accurate 시뮬레이터 기반(실 hardware prototype 부재) — 절대 수치 신뢰도는 모델 정확도에 의존.
- SSD DRAM이 high throughput에서 병목이라고 저자 스스로 인정(§VIII); HBM/DRAM bypass는 future work.
- DirectGraph는 static graph 가정 — dynamic graph(빈번한 edge 변경)에는 재구성 비용이 클 수 있음.
- OGBN 같은 low-degree 그래프는 inflate 32.3%로 공간 오버헤드가 큼.
- 주로 GNN training 대상; latency-critical real-time query는 §VIII에서 언급만.

**열린 질문**
- 실제 ULL SSD 프로토타입에서 die-level sampler의 timing/공정 budget이 정말 <0.1%로 들어가는가?
- multi-SSD storage array P2P 협업(§VIII)에서 DirectGraph의 physical address가 SSD 경계를 넘을 때 어떻게 처리되나?
- write-heavy/dynamic GNN에서 DirectGraph reconstruction + reclamation의 amortized 비용은?

## ❓ Q&A (자가 점검)

> [!question]- Q1. 대규모 GNN에서 data preparation이 병목인 이유는?
> > graph 구조와 feature table이 TB급이라 SSD에 상주하고, neighbor sampling·feature lookup이 thin PCIe bus 너머로 대량 전송을 유발하며 file system+NVMe stack이 redundant copy와 다중 주소 변환을 더하기 때문. (p.1)

> [!question]- Q2. 기존 ISC-for-GNN(GList, SmartSage)의 세 결함은?
> > (1) hop-by-hop in-order 샘플링으로 flash 내부 병렬성 미활용(inter-hop barrier), (2) page보다 작은 random I/O를 page-granular channel 전송으로 처리해 대역폭 낭비, (3) firmware-기반 I/O 처리가 ULL flash throughput을 못 따라감. (Abstract, §III, p.1·3~4)

> [!question]- Q3. DirectGraph가 해결하는 것과 핵심 아이디어는?
> > GNN 데이터가 static이라는 관찰을 이용해 neighbor index를 4-byte flash physical address로 직접 박고 primary/secondary section을 physical page에 정렬. 다단 주소 변환과 hop마다의 host-SSD 통신(barrier)을 제거해 SSD 내부에서 out-of-order 샘플링 가능. (§IV-A, p.5)

> [!question]- Q4. 3단 near-data 엔진 각각의 역할은?
> > die-level sampler: page 내 neighbor 샘플링+feature retrieve+다음 command 생성(firmware 없이). channel-level command router: sampling command를 crossbar로 destination channel에 라우팅(embedded processor 우회). controller-level spatial accelerator: 1D vector array aggregation + 2D systolic GEMM update. (§V, p.6~7)

> [!question]- Q5. BG-1과 BG-2의 차이는?
> > BG-1(BeaconGNN-1.0)은 GList+SmartSage를 합쳐 전체 task를 SSD로 오프로드+ULL flash 적용한 baseline 조합이나 세 challenge가 그대로 남음. BG-2는 DirectGraph + die/channel/controller 다단 엔진으로 세 challenge를 모두 해결. (Fig.4 vs Fig.9, p.4~5)

> [!question]- Q6. 핵심 성능·에너지 수치는?
> > CPU-centric baseline 대비 최대 27.3× throughput·9.86× energy, SOTA ISC(BG-DGSP) 대비 throughput 11.6×(overall)·에너지 4×. BG-2는 BG-DGSP 대비 +41%, channel·die 활용 76%↑, sampling latency 78%↓, command wait 68%↓. (p.2·8~10)

> [!question]- Q7. 추가한 하드웨어의 area/power 오버헤드는?
> > channel router는 SSD controller에 1.26% area, sampler+config+TRNG는 ULL die의 <0.1%. 평균 power 13.4W로 PCIe 75W 한계 이하. (§VII-A, §VII-D, p.8·12)

> [!question]- Q8. BeaconGNN의 알려진 한계는?
> > high throughput에서 SSD DRAM 대역폭이 병목(저자 인정, §VIII), DirectGraph가 static graph 가정이라 dynamic graph에 약함, OGBN 같은 low-degree 그래프는 inflate 32.3%, 평가가 시뮬레이션 기반. (§VII-F·§VIII)
